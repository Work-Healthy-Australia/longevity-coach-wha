Option 1 — recommended

  • Fixes: per-relative ages preserved · `multiple` flag derivable · `age_onset` no longer ambiguous
  • Mirrors the existing `cancer_history` typed-field pattern — proven UX, proven hydration, proven engine path
  • Engine code untouched; only `assemble.ts` changes (~10 lines)
  • Same relative can repeat (two aunts, three cousins) — no false-uniqueness constraint
  • 4 new typed-field renderers vs. 8 flat fields today (small UI complexity bump)
  • Hydration shim needed once for old responses

Effort: ~1 day · Affects: family step only · Migration: hydration shim, no DB change

Option 2 — Base44-style invert

  • Closer to clinical mental model ("my mom had X, Y, Z")
  • Captures vital status + cause of death + smoking + alcohol per relative — collapses two existing steps into one
  • Ready for the future "longevity actuarial" model that needs deceased ages and causes
  • Largest blast radius — touches family-history AND deaths step; new schema; more renderer work
  • Hydration shim needs to merge old condition data + old deaths-step data into the new per-relative shape
  • Engine assemble logic gets more complex (denormalise per-relative into per-condition aggregates)

Effort: ~3 days · Affects: family + deaths steps · Migration: complex hydration shim, no DB change

Option 3 — two-step disclosure (lightest)

  • Smallest schema change — keeps the multiselect, replaces the single age field with a typed map
  • Familiar UX (multiselect first, age inputs reveal as you toggle)
  • Two coupled fields per condition — harder to validate consistently (orphaned ages when you uncheck a relative)
  • Same relative can't be added twice (two aunts? two cousins?) — capped at 9 relative types
  • Awkward as relatives toggle on/off — age inputs flash in/out of existence
  • Doesn't easily accommodate the future per-relative-card refactor

Effort: ~0.5 day · Affects: family step only · Not recommended — fixes everything Option 1 fixes but with worse UX
