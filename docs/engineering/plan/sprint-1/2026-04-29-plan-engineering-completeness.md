## Wave 2 — Agent Suite Completion

### W2-1: PT Coach agent — pipeline worker (monthly plan)

**What:** Build the monthly PT plan pipeline worker at `lib/ai/pipelines/pt-plan.ts`.

Design follows the same pattern as Atlas (`lib/ai/pipelines/risk-narrative.ts`):
- Input: user UUID → reads `PatientContext` (risk scores, questionnaire, daily_logs last 30 days)
- Prompt: generate a structured 30-day progressive exercise program personalised to:
  - MSK risk domain score and specific risk drivers
  - Current activity level (from questionnaire)
  - Step count trends (from `biomarkers.daily_logs`)
  - Age, weight (derived from profile)
- Output schema: `{ program_name, weekly_structure, session_count_per_week, notes, generated_at }`
- Write target: `programs.pt_plans` (table exists in migration `0012_programs_schema.sql` — verify column fit; add migration if schema needs extension)
- Idempotent upsert keyed on `(user_uuid, plan_start_date)`
- Triggered by: Atlas completion (same trigger chain as Sage) + monthly cron

Implement:
- `lib/ai/pipelines/pt-plan.ts`
- `app/api/pipelines/pt-plan/route.ts` — secured with `x-pipeline-secret`

### W2-9: Janet-Clinician Brief pipeline worker (Epic 9 agent)

**Architecture reference:** `docs/architecture/clinician-portal.md` — read this before writing any code. The Base44 prototype built a working `janet_clinician` agent; Section 1.3 and Section 5 capture every proven design decision.

**What this step builds:** The pipeline worker (`janet_brief` pre-generation) only. The portal UI workspace is blocked on James's review session (Plan A Step 13). The brief pipeline can and should ship independently — clinicians don't need the portal to exist for `periodic_reviews` rows to start accumulating.

**Design (from architecture doc Section 1.3 + Section 5):**
- Execution model: pipeline worker (monthly cron), not real-time — produces `janet_brief` before the clinician opens the workspace so they see the summary instantly with zero LLM wait
- Input: `patient_uuid` + `clinician_uuid` → reads: `CheckinReview`/`periodic_reviews`, `PatientContext` (risk scores, supplement plan, lab results, last 30 days of check-ins, `member_alerts`, last `coach_suggestions`)
- Persona: Janet speaking *to a clinician colleague* — professional, evidence-based, concise (distinct from member-facing Janet tone)
- Output schema (maps to `periodic_reviews`):
  ```typescript
  {
    janet_brief: string,          // structured patient summary shown in Patient Card tab
    domain_highlights: DomainHighlight[],
    suggested_focus: string[],    // one per risk domain, evidence-grounded
    adherence_signals: string,    // check-in consistency summary
    biomarker_deltas: string,     // notable changes since last brief
    alert_summary: string,        // outstanding member_alerts
    data_coverage_note: string,   // what data was available vs. missing
    generated_at: string,
  }
  ```
- Write target: `clinical.periodic_reviews` — idempotent upsert keyed on `(patient_uuid, review_month)` where `review_month = date_trunc('month', now())`
- `review_status` initial value: `awaiting_clinician` (status enum from Section 2: `awaiting_clinician → in_review → program_ready → sent_to_patient`)
- Triggered by: 1st of each month cron per active `patient_assignments` row

**Schema prerequisites** (check before coding):
- `clinical.periodic_reviews` exists (migration `0011`) — verify it has columns for `janet_brief`, `review_status`, `clinician_conversation_id`, `program_30_day`. Add migration `0047_periodic_reviews_expand.sql` if columns are missing.
- `profiles.role` check constraint still only allows `user` and `admin` — the portal UI migration will expand this, but the pipeline worker only reads `patient_assignments` so it does not need the role expansion yet.

**Implement:**
- `lib/ai/pipelines/clinician-brief.ts`
- `app/api/pipelines/clinician-brief/route.ts` — secured with `x-pipeline-secret`; accepts `{ patient_uuid, clinician_uuid }`
- `app/api/cron/clinician-briefs/route.ts` — monthly cron; iterates all active `patient_assignments` rows; fires one pipeline call per assignment; returns 200 on any individual failure to prevent Vercel retry storms
- Register `0 1 1 * *` (1AM on the 1st of each month) in `vercel.json`
- Unit tests: `tests/unit/ai/clinician-brief-helpers.test.ts`
- Integration test: `tests/integration/ai/clinician-brief.test.ts` (mocked Supabase + one seeded `patient_assignments` row)

**What the portal UI needs additionally (deferred to after Plan A Step 13 review):**
- `clinician_profiles` table (new — absent from production schema, Section 6)
- `appointments` table (new — absent from production schema, Section 6)
- `profiles.role` constraint expansion to include `clinician`, `coach`, `health_manager` (Section 6: role system delta)

## Wave 3 — Epic 7 Daily Return Features

### W3-1: Personalised daily goals

**What:** Replace static goal defaults in `/check-in` with risk-profile-derived targets.

Logic in `lib/goals/derive.ts`:
- Steps target: if MSK risk `high` → 6000; if CV risk `high` → 10000; default 8000
- Sleep target: if brain risk `high` → 8h; default 7.5h  
- Water target: body weight × 0.033L (from questionnaire weight field); default 8 glasses
- Meditation: if stress/anxiety indicators in questionnaire → 10 min daily; else optional

Store derived goals in `profiles` as a `daily_goals JSONB` column (new migration `0044_daily_goals.sql`). Recomputed by Atlas pipeline on each run (write after risk score update).

Display on `/check-in` as personalised targets with one-line rationale ("based on your cardiovascular risk").

Unit tests for each derivation rule.
